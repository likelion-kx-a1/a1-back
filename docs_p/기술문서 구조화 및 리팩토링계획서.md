🪐 기술 문서(Technical Documentation) 구조화 및 리팩토링 계획서

이 문서는 프로젝트 내의 혼잡한 docs_h/ 폴더를 정리하여 실제 현업 프로덕션에 준하는 깔끔한 아키텍처 문서 저장소로 전환하고, AI 프롬프트 도구들을 별도의 저장소로 격리하기 위한 종합 가이드라인입니다.

📊 1. 상용 프로젝트의 표준 기술 문서 구성 요소

일반적으로 잘 관리된 백엔드/풀스택 프로젝트의 문서 저장소는 아래 5대 영역을 반드시 포함하며, 중복이나 임시 가이드는 최소화합니다.

시작 가이드 (Onboarding / Setup):

새로 합류한 개발자가 자신의 로컬 컴퓨터에 Docker, Database, Node.js, Spring Boot 환경을 세팅하고 실행하는 단 하나의 정석 매뉴얼.

아키텍처 및 도메인 구조 (Architecture & Directory Layout):

시스템의 전체 데이터 흐름(Data Flow), DDD 패키지 구조 설계 원칙, 레이어 간 호출 규칙(Port/Adapter) 정리.

API 명세서 (API Specification):

프론트엔드와 백엔드가 통신하는 모든 엔드포인트 주소, HTTP 메서드, 요청 DTO, 응답 JSON 스키마 명세.

테스트 및 시뮬레이션 매뉴얼 (Testing & Mocking Guide):

로컬 모킹(Mocking) 환경에서 비용 없이 테스트하는 방법 및 배포 시 실제 상용(Prod) 인프라로 스위칭하는 방법.

트러블슈팅 가이드 (Troubleshooting Master):

CORS, 404 Not Found, DB 패스워드 인증 실패, 400 Bad Request 등 자주 만나는 에러 코드별 진단법과 처방전.

🛠️ 2. 리팩토링 Target 폴더 구조 설계안

수많은 개별 파일들을 역할에 맞추어 하나로 병합(Consolidation)하고, 프롬프트 파일들은 루트의 prompts/ 폴더로 격리합니다.

a1-back/
├── docs_p/                              # AI 코드 작성을 위해 사용했던 프롬프트 도구 격리
│   ├── antigravity_prompt_engine.md      # 안티그래비티 물리-카메라 시네마틱 영/한 엔진 명세
│   ├── claude_backend_generation.md      # Spring Boot 소스코드 자동 생성용 마스터 프롬프트
│   └── ddd_scaffolding_prompt.md         # DDD 패키지 구조 설계용 프롬프트
│
└── docs_h/                               # 진짜 협업용 실무 기술 문서 저장소
├── api_specification.md              # [통합] 최종 API 명세서 (최신 v2.0 규격 반영)
├── architecture_guide.md             # [통합] DDD 하이브리드 패키지 설계 및 데이터 흐름도
├── local_setup_guide.md              # [통합] PostgreSQL Docker + Next.js 올인원 온보딩 매뉴얼
├── troubleshooting_guide.md          # [통합] CORS, 404, 400, DB인증 에러 해결 조치서
└── testing_and_simulation_guide.md   # [통합] 비용 $0 로컬 Mock 시뮬레이터 및 상용망 배포 전환 가이드


영구 삭제 대상: project_outsourcing_cost_estimation.md (외주 견적용 임시 문서), file_mapping_guide.md (임시 매핑 정보) 등 개발에 기여하지 않는 문서들은 아카이빙 후 제거합니다.

📋 3. 클로드(Claude Code CLI / Chat) 주입용 자동 정리 프롬프트

아래 점선 안쪽의 내용을 통째로 복사하여 현재 터미널의 Claude Code 또는 대화 중인 클로드 창에 입력하면, 클로드가 실제 소스코드와 명세서를 리딩하여 위 구조대로 모든 파일을 자동으로 리팩토링(병합 및 정돈)해 줍니다.

역할: Senior Technical Writer & System Architect

목표:
현재 프로젝트 내의 혼잡하게 분산된 기술 문서들을 표준 규격에 맞추어 유기적으로 병합하고 정리하라. 중복되는 자잘한 트러블슈팅과 가이드라인을 하나의 완성도 높은 마스터 기술 문서 세트로 리팩토링하고, AI 에이전트 생성용 프롬프트들은 프로젝트 루트의 prompts/ 디렉토리로 안전하게 격리하라.

AI 프롬프트 격리 (prompts/ 생성):

안티그래비티 물리 엔진 프롬프트 관련 지침(antigravity_prompt_system.md, antigravity_prompt_system_ko.md)을 하나로 완벽히 다듬어 prompts/antigravity_prompt_engine.md에 보관할 것.

Spring Boot 자동 생성용 지침(claude_implementation_prompt.md, refactored_claude_implementation_prompt.md, llm_generation_prompt.md)을 병합하여 prompts/claude_backend_generation.md로 정리할 것.

구조 설계 프롬프트(package_structure_prompt.md, api_spec_generation_prompt.md)는 prompts/ddd_scaffolding_prompt.md로 정리할 것.

기존에 소모된 원본 프롬프트 파일들은 모두 깔끔하게 디스크에서 삭제할 것.

코어 기술 문서 통합 및 정돈 (docs_h/ 재정비):

docs_h/api_specification.md: 사용자가 대화창에 제공했던 최신 "a1-back Generation API 명세서 v2.0" 텍스트를 기반으로 명세서 파일을 신규 생성 혹은 갱신할 것.

docs_h/architecture_guide.md: 기존 current_project_status_guide.md 내용을 근간으로 하여 패키지 레이아웃과 도메인 분리 가이드를 유기적으로 병합하여 생성할 것.

docs_h/local_setup_guide.md: local_dev_environment_setup_guide.md와 frontend_setup_and_conceptual_guide.md 내용을 병합하여, Docker 데이터베이스 구동부터 Next.js 연결까지 0부터 끝까지 따라 할 수 있는 정석 온보딩 가이드라인으로 완성할 것.

docs_h/troubleshooting_guide.md: db_connection_error_fix.md, cors_and_network_error_fix.md, api_endpoint_check_guide.md, api_404_troubleshooting.md, http_error_troubleshooting_guide.md에 흩어져 있는 CORS, 404, 400, DB인증 에러 조치사항을 일목요연한 목차로 하나로 통합하여 구축할 것.

docs_h/testing_and_simulation_guide.md: local_dev_test_guide.md, local_api_test_and_frontend_integration_guide.md, testing_and_cost_saving_guide.md, web_server_and_prod_transition_guide.md 내용을 통합하여, 가짜 어댑터 작동 매커니즘, 포스트맨 검증법, Next.js 비동기 폴링 리액트 예시코드, 그리고 상용망 전환 시 Active Profile 제어법을 하나의 가이드로 정밀화할 것.

기타 transient 임시 파일 청소:

더 이상 쓸모가 없어진 임시 파일(file_mapping_guide.md, terminal_agent_guide.md 등)은 깔끔히 삭제 처리할 것.

이 리팩토링 지침을 한치의 흐트러짐도 없이 파악했다면 즉시 파일 통합 및 폴더 격리 작업을 완료해 주길 바란다.
