# a1-back 백엔드 아키텍처 가이드 (DDD 하이브리드 계층형)

이 문서는 `a1-back` 프로젝트가 채택한 **DDD(도메인 주도 설계) + 계층형(Layered) 하이브리드 아키텍처**의 설계 배경, 실제 적용 결과, 그리고 새 도메인/기능을 추가할 때 따라야 할 규칙을 정리합니다.

---

## 1. 배경 — 왜 순수 헥사고날을 걷어냈는가

프로젝트 초기에는 일부 도메인(`user`, `generation`)이 **헥사고날 아키텍처(Ports & Adapters)**를 충실히 적용하고 있었습니다.

```
Service → domain.repository.UserRepository (Port 인터페이스)
        → infrastructure.persistence.UserRepositoryAdapter (Adapter 구현체)
        → infrastructure.persistence.SpringDataUserRepository (Spring Data JPA 인터페이스)
```

MVP 단계의 빠른 기능 개발 앞에서 이 구조는 다음 한계를 보였습니다.

- **과도한 보일러플레이트**: 단순 CRUD 하나를 추가할 때도 `Repository Interface` + `Spring Data JPA Interface` + `Adapter 구현체` 3개를 매번 작성해야 함.
- **인터페이스 무용화**: 거의 모든 도메인 JPA 레포지토리가 1:1 어댑터만 가지고 있어, 실제 추상화 효용 없이 계층만 통과하는 코드가 됨.

이에 따라 **Pragmatic Hybrid Layered DDD**로 단순화하기로 결정했습니다.

---

## 2. 하이브리드 아키텍처 핵심 규칙

1. **JPA Entity = Domain Model 통합** — 순수 도메인 객체와 JPA 엔티티를 분리하지 않고 하나로 통합해 매퍼(Mapper) 레이어를 없앤다. 엔티티는 도메인 하위의 `domain/model`에 두고 JPA 어노테이션을 직접 사용한다.
2. **도메인 내부 4계층 구조**:
   - `presentation`: Controller, DTO (외부 클라이언트 경계)
   - `application`: Service (비즈니스 오케스트레이션, 트랜잭션 경계), 외부 연동 포트 인터페이스(`port/out`)
   - `domain`: 핵심 비즈니스 로직(Entity)
   - `infrastructure`: Spring Data JPA Repository, 외부 API 클라이언트 구현체
3. **DB CRUD와 외부 API 연동은 다르게 취급한다** (아래 4장 참고) — 이 구분이 이 아키텍처의 핵심 판단 기준이다.

## 3. DB 영속성 계층 — Port/Adapter 제거, Spring Data JPA 직접 사용

`user`, `generation` 등 **단순 CRUD 위주 도메인**은 Port 인터페이스와 Adapter 구현체를 전면 제거하고, `application.service`가 Spring Data JPA Repository를 직접 주입받아 사용한다.

### Before (헥사고날 보일러플레이트)

```text
com/likelion/a1/
├── user
│   ├── application/service/UserCommandService.java   # UserRepository(Port) 의존
│   ├── domain/repository/UserRepository.java         # [Port 인터페이스]
│   └── infrastructure/persistence/
│       ├── SpringDataUserRepository.java             # [Spring Data JPA 인터페이스]
│       └── UserRepositoryAdapter.java                 # [Adapter 구현체]
└── generation
    ├── domain/repository/GenerationJobRepository.java # [Port 인터페이스]
    └── infrastructure/persistence/
        ├── SpringDataGenerationJobRepository.java     # [Spring Data JPA 인터페이스]
        └── GenerationJobRepositoryAdapter.java         # [Adapter 구현체]
```

### After (적용 완료, 실제 현재 구조)

```text
com/likelion/a1/
├── user
│   ├── application/service/UserCommandService.java   # UserRepository를 직접 의존
│   └── infrastructure/persistence/
│       └── UserRepository.java                       # [통합 Spring Data JPA 리포지토리 인터페이스]
└── generation
    └── infrastructure/persistence/
        └── GenerationJobRepository.java              # [통합 Spring Data JPA 리포지토리 인터페이스]
```

**삭제된 파일 (6개)**: `user/domain/repository/UserRepository.java`, `user/infrastructure/persistence/{UserRepositoryAdapter,SpringDataUserRepository}.java`, `generation/domain/repository/GenerationJobRepository.java`, `generation/infrastructure/persistence/{GenerationJobRepositoryAdapter,SpringDataGenerationJobRepository}.java`

**신규 통합 파일 (2개)**: `user/infrastructure/persistence/UserRepository.java`, `generation/infrastructure/persistence/GenerationJobRepository.java` (둘 다 `JpaRepository` 직접 상속)

### 새 도메인 추가 시 절차

```java
// src/main/java/com/likelion/a1/user/infrastructure/persistence/UserRepository.java
package com.likelion.a1.user.infrastructure.persistence;

import com.likelion.a1.user.domain.model.User;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;

@Repository
public interface UserRepository extends JpaRepository<User, Long> {
  boolean existsByEmailIgnoreCase(String email);
}
```

```java
// src/main/java/com/likelion/a1/user/application/service/UserCommandService.java
@Service
public class UserCommandService {
  private final UserRepository repository; // 바로 사용, 중간 Adapter 없음

  public UserCommandService(UserRepository repository) {
    this.repository = repository;
  }

  @Transactional
  public User register(String email, String password, String nickname) {
    if (repository.existsByEmailIgnoreCase(email)) {
      throw new BusinessException(ErrorCode.USER_EMAIL_DUPLICATE);
    }
    return repository.save(User.local(email, password, nickname));
  }
}
```

## 4. 외부 API 연동 계층 — Port/Adapter는 그대로 유지

⚠️ **3장의 단순화는 DB CRUD 레이어에만 적용된다.** AWS S3 스토리지 연동, 외부 AI API(Claude, GPT Vision, fal.ai) 클라이언트 연동처럼 **SDK/HTTP Client 세부 기술에 종속성이 크고 로컬 Mock 테스트 격리가 필수적인 아웃바운드 인프라**는 `application/port/out` (인터페이스) + `infrastructure/client` (구현체) 형태를 그대로 유지한다.

이렇게 나누는 이유: 외부 API는 로컬 개발 중 Mock 구현체로 완전히 대체 가능해야 하기 때문에(비용 절감, API 키 없이 개발), 인터페이스 경계가 실질적인 가치를 가진다. 반면 DB Repository는 테스트 시 실제 DB(Testcontainers 등)를 쓰는 편이 낫고 어차피 구현체가 하나뿐이라 인터페이스가 무의미했다.

### `generation` 도메인의 AI Provider 포트 3종

| Port (`application/port/out`) | Mock 어댑터 (`@Profile("local")`) | 실제 어댑터 (`@Profile("!local")`) |
|---|---|---|
| `PromptGenerationPort` | `claude/MockPromptGenerationAdapter` | `claude/ClaudePromptGenerationAdapter` (Anthropic Messages API) |
| `ImageAnalysisPort` | `openai/MockImageAnalysisAdapter` | `openai/GptVisionImageAnalysisAdapter` (OpenAI Chat Completions) |
| `FalGenerationPort` | `fal/MockFalGenerationAdapter` | `fal/FalGenerationAdapter` (fal.ai 큐 API: submit + poll) |

결과 타입(`AiTextGenerationResult`, `FalGenerationSubmission`, `FalGenerationStatus`)은 원본 응답(`rawResponse`)을 그대로 들고 있어 `generation_jobs.response_payload`(JSONB)에 그대로 저장할 수 있다.

세 어댑터는 각기 다른 인증 헤더를 쓰며, `RestClient` Bean 3개(`falRestClient`/`claudeRestClient`/`openAiRestClient`)는 `global/config/AiClientConfig.java`에 정의되어 있다:
- fal.ai: `Authorization: Key <FAL_API_KEY>`
- Claude: `x-api-key: <ANTHROPIC_API_KEY>` + `anthropic-version` 헤더
- OpenAI: `Authorization: Bearer <OPENAI_API_KEY>`

Mock/실제 전환 방법과 각 Mock의 동작 방식은 `테스트_및_시뮬레이션_가이드.md`를 참고한다.

### `media` 도메인의 스토리지 포트

- `application/port/out/MediaStoragePort.java` — S3 등 물리 저장소에 저장하기 위한 아웃풋 포트.
- `infrastructure/storage/s3/S3MediaStorageAdapter.java` — AWS S3 SDK로 실제 업로드를 수행하는 어댑터.
- `infrastructure/storage/local/MockMediaStorageAdapter.java` — 로컬 프로필용 더미 스토리지 어댑터.

여러 도메인에서 공통으로 S3를 써야 할 필요가 커지면, 이 포트/어댑터를 `global` 수준으로 상향하는 것을 향후 검토한다.

## 5. `GenerationJob` — 단일 엔티티 + 상태 전이

`generation` 도메인은 이미지/영상 생성 요청 4종(프롬프트 보정, 역프롬프트, fal.ai 제출, fal.ai 폴링)을 모두 `GenerationJob` 엔티티 하나로 표현한다 (별도의 `ImageAsset`/`VideoTask` 테이블로 나누지 않음).

- `GenerationJob.create(userId, chatId, aiModelId, requestMessageId, jobType, prompt, requestPayload)` — 최초 저장용 정적 팩토리. 생성 시 상태는 `PROCESSING`.
- `markQueued(...)`, `complete(...)`, `fail(...)`, `applyPolledStatus(...)` — 상태 전이 메서드.
- `GenerationAiService`는 각 메서드에서 **"저장(PROCESSING) → 외부 API 호출(트랜잭션 밖) → 결과 저장"** 순서로 동작한다. 메서드 전체를 `@Transactional`로 감싸지 않는데, 이는 "외부 API 호출 중 DB 트랜잭션을 유지하지 않는다"는 요구사항을 따른 것이다. 실패 시에도 예외를 다시 던지기 전에 `FAILED` 상태로 먼저 저장한다.
- `pollFalJob(jobId)`는 jobId 하나만 받아 저장된 `requestPayload.modelCode` / `responsePayload.requestId`를 꺼내 fal.ai에 재조회한다.
- `GenerationVideoPollingScheduler`가 5초 주기로 `QUEUED`/`PROCESSING` 상태 job을 스캔해 자동으로 폴링하고, 완료 시 `MediaStoragePort`로 업로드해 `responsePayload.s3Url`을 채운다.

> **아직 미완/알려진 제약**: `aiModelId`는 항상 `null`(`ai_models` 테이블 연동 로직 없음), fal.ai 완료본을 `generated_media`/`generated_media_files`로 이관하는 로직 없음, 이미 `COMPLETED`/`FAILED`인 job을 다시 poll해도 막지 않음(상태 전이 검증 미구현).

- **컴파일/테스트**: `./gradlew test`로 Spring Boot 컨텍스트 정상 로드 및 SQL 바인딩 검증 완료.
- **1단계 (완료)**: DB CRUD 도메인(`user`, `generation`)의 Port/Adapter 보일러플레이트 제거.
- **2단계 (완료)**: 외부 AI 3사(Claude/GPT Vision/fal.ai) 연동을 `application/port/out` + `infrastructure/client` 패턴으로 스캐폴딩, Mock/실제 어댑터 쌍으로 분리.
