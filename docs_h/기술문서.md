## 1. 전체 아키텍처 요약

패키지 구조는 도메인별 계층형(Layered) 구조를 따른다.

```
com.likelion.a1
├── global        # 공통 설정/예외/응답 포맷 (도메인 아님)
├── user          # 인증, 회원가입/승인, 관리자 회원 관리
├── project       # 프로젝트(작업 폴더) CRUD
├── chat          # 채팅방 + 채팅 메시지 + 메시지 첨부파일
├── media         # 파일 업로드(S3), 생성 결과물(GeneratedAsset) 저장
├── generation    # (스키마/엔티티만 존재, 실질적으로 미사용 — 7장 참고)
├── notification  # 엔티티/DTO만 존재, 서비스/컨트롤러 없음
└── library       # package-info만 존재, 실질적으로 빈 껍데기
```

각 도메인 내부는 `presentation`(Controller/DTO) → `application`(Service) → `domain`(Entity/Repository 인터페이스) → `infrastructure`(Spring Data JPA 구현, 외부 클라이언트)의 4계층을 따르는 것이 원칙이나, **실제로는 도메인마다 패턴이 다르다** (아래 2장, 9장 참고).

### 요청 처리 흐름 (공통)

1. 모든 요청은 `JwtAuthenticationFilter`(OncePerRequestFilter)를 거친다. `Authorization: Bearer <token>` 헤더가 있으면 `JwtTokenProvider.parse()`로 검증 후 `JwtPrincipal`(userId/loginId/role/sessionId)을 `SecurityContext`에 채운다. 토큰이 없거나 파싱 실패 시 그냥 익명으로 통과시킨다(예외를 던지지 않고 컨텍스트만 비움).
2. `SecurityConfig`가 URL 단위로 인가를 결정한다.
   - `permitAll`: `/actuator/health/**`, `/api/auth/check-login-id`, `/api/auth/email/send`, `/api/auth/email/verify`, `/api/auth/signup`, `/api/auth/login`, `/api/auth/refresh`, `/api/auth/password/reset`
   - `/api/admin/**`: `JwtPrincipal.role() == "ADMIN"`인 경우만 허용
   - 나머지 전부: 인증 필요 (`anyRequest().authenticated()`)
   - CSRF 비활성화, 세션 `STATELESS`, CORS는 `app.cors.allowed-origin-patterns`(기본값 localhost:3000/5173 + `*.vercel.app`) 기준으로 허용
3. 컨트롤러는 `@AuthenticationPrincipal JwtPrincipal principal`로 `userId`를 꺼내 서비스에 전달한다. (예전 문서에 있던 "userId를 요청 바디로 직접 받는 임시 스펙"은 `chat`/`project`/`media` 도메인에서는 이미 JWT 기반으로 대체되어 있다.)
4. 성공 응답은 `ApiResponse<T>(success, code, message, data, timestamp)`, 실패는 `ErrorResponse(success=false, error{code,message,details}, timestamp)`로 통일되어 있다. 단, `GlobalExceptionHandler`가 처리하는 예외는 `BusinessException`과 `MethodArgumentNotValidException` **두 가지뿐**이다 (9.5절 참고).

## 3. `user` 도메인 — 인증/회원가입/관리자

### 3.1 회원가입 & 로그인 (`AuthController`, base: `/api/auth`)

- `GET /check-login-id` — 로그인 아이디 중복 확인 (permitAll)
- `POST /email/send`, `POST /email/verify` — 이메일 인증번호 발송/검증 (permitAll). 인증번호는 원문을 저장하지 않고 SHA-256 해시만 `email_verifications` 테이블에 저장하며, **5분** 후 만료된다.
- `POST /signup` — 회원가입 신청. `loginId`/`email` 중복 검사 후, 해당 이메일이 `SIGNUP` 목적으로 이메일 인증 완료(`validateVerified`) 상태여야 통과한다. 가입 즉시 `approvalStatus=PENDING`, `accountStatus=INACTIVE`로 저장되며 **로그인은 관리자 승인 전까지 불가능**하다.
- `POST /login` — `loginId` + 비밀번호 검증 후 `approvalStatus=APPROVED && accountStatus=ACTIVE`인 경우에만 로그인 허용(`isLoginAllowed()`). 성공 시 `AuthSession`(세션ID, refresh token 해시, IP, User-Agent, 14일 만료)을 저장하고 JWT Access Token(기본 3600초)을 발급한다.
- `POST /refresh` — refresh token 해시로 세션 조회 후 access token 재발급.
- `POST /logout` — 세션 폐기(`revoke`) + `lastLogoutAt` 기록.
- `POST /password/reset` — 이메일 인증(purpose=`PASSWORD_RESET`) 확인 후 비밀번호 변경, 해당 유저의 모든 세션을 일괄 폐기(`revokeAllByUserId`).

### 3.2 관리자 회원 관리 (`AdminUserController`, base: `/api/admin/users`, `role=ADMIN` 필요)

- `GET /signup-requests`, `GET ?approvalStatus=&accountStatus=&keyword=` — 페이지네이션 목록 조회 (`Pageable`)
- `GET /{userId}` — 상세 조회
- `PATCH /{userId}/approve` — 승인 처리. 승인 시 **트랜잭션 커밋 이후**(`TransactionSynchronization.afterCommit`)에 회원가입 승인 메일을 비동기 성격으로 발송하고, 메일 발송 실패는 로그만 남기고 승인 자체는 롤백시키지 않는다.
- `PATCH /{userId}/reject` — 거절 처리 (사유 필수)
- `PATCH /{userId}/status` — 계정 상태(`ACTIVE`/`INACTIVE`) 변경
- `DELETE /{userId}` — 소프트 삭제(`deletedAt` 세팅, 실제 row 삭제 아님)
- 승인/거절/상태변경/삭제 모두 "관리자가 자기 자신을 대상으로 조작할 수 없다"(`ADMIN_CANNOT_UPDATE_SELF`)는 규칙이 있다.

### 3.3 이메일 발송 (`EmailSenderPort` / `GmailEmailSenderAdapter`)

Gmail SMTP(`spring.mail.*`)를 사용하는 어댑터 하나만 존재한다. `@Profile` 분기나 로컬용 Mock 구현체가 없으므로, **로컬 개발 환경에서도 실제 SMTP 계정(`MAIL_USERNAME`/`MAIL_PASSWORD`) 설정이 필요**하다 (설정이 없으면 발송 시 예외 발생 — 단, 승인 메일은 실패해도 로그만 남기므로 승인 자체는 성공한다).

### 3.4 레거시/사용되지 않는 경로 — `UserController` / `UserCommandService`

`/api/v1/users` (`UserController.register`)와 이를 지원하는 `UserCommandService`, `user.infrastructure.persistence.UserRepository`(Spring Data JPA 직접 상속)는 `AuthController`/`AuthService` 기반 회원가입 체계와는 **완전히 별개의, 더 이상 쓰이지 않는 구현**이다. 자세한 내용은 9.2절 참고.

## 4. `project` 도메인

- `ProjectController` (base `/api/projects`, 인증 필요): 생성/목록/단건조회/수정/삭제(soft delete).

## 5. `chat` 도메인 — 채팅방/메시지/첨부파일

### 5.1 엔티티

- `Chat`: `userId`, `projectId`(nullable — null이면 프로젝트 밖 독립 채팅방), `title`, `generationType`(`IMAGE`/`VIDEO`), `imageCategory`, `firstMessageId`, `isGenerating`(생성 작업 진행 플래그), `status`(`ACTIVE`/`DELETED`).
- `ChatMessage`: `chatId`, `userId`, `senderType`(`USER`/`ASSISTANT`), `messageType`(`TEXT`/`FILE`/`MIXED`/`ASSET`/`ASSET_REQUEST` 등), `contentText`, `generationType`, `imageCategory`, `parentMessageId`, `generationJobId`, `generatedAssetId`, `sortOrder`, `status`.
- `ChatMessageFile`: 메시지에 첨부된 파일의 메타데이터(버킷/경로/공개 URL/원본·저장 파일명/mimeType/크기/가로세로/재생시간).

### 5.2 API (`ChatController`, `ChatMessageController`, `FileUploadController`)

| Method | Path | 설명 |
|---|---|---|
| POST | `/api/chats` | 채팅방 생성(프로젝트 미지정 가능) |
| POST | `/api/projects/{projectId}/chats` | 특정 프로젝트 하위 채팅방 생성 |
| GET | `/api/chats?projectId=&outsideProject=` | 채팅방 목록 (조건부 필터) |
| GET | `/api/chats/standalone` | 프로젝트 밖 채팅방 목록 |
| GET | `/api/projects/{projectId}/chats` | 프로젝트 하위 채팅방 목록 (위 GET과 기능 중복, 9.7절 참고) |
| GET | `/api/chats/{chatId}` | 채팅방 상세(메시지+첨부파일 포함) |
| PATCH | `/api/chats/{chatId}` | 제목 수정 |
| DELETE | `/api/chats/{chatId}` | 소프트 삭제 |
| POST | `/api/chats/{chatId}/messages` | 메시지 생성 (JSON, 첨부는 이미 업로드된 파일 메타데이터 배열로 전달) |
| POST | `/api/chats/{chatId}/messages` (multipart) | 메시지 생성 + 파일 동시 업로드 (`Content-Type: multipart/form-data`로 구분) |
| GET | `/api/chats/{chatId}/messages` | 메시지 목록 |
| PATCH | `/api/chats/{chatId}/messages/{messageId}` | 메시지 내용 수정 |
| DELETE | `/api/chats/{chatId}/messages/{messageId}` | 메시지 소프트 삭제 |
| POST | `/api/chats/{chatId}/files` | 단일 파일 업로드(메시지와 별개, S3 저장 후 메타데이터만 반환) |

### 5.3 메시지 생성 검증 로직 (`ChatMessageService`)

- 텍스트도 파일도 없으면 거부.
- `messageType`이 `IMAGE`/`VIDEO`/`ASSET`/`ASSET_REQUEST`면 `generationType` 필수.
- `generationType=IMAGE`일 때 `imageCategory`는 `CHARACTER`/`BACKGROUND`/`ETC` 중 하나만 허용, `generationType=VIDEO`일 때는 `imageCategory`가 있으면 안 됨.
- `messageType=TEXT`인데 텍스트 없음 / `messageType=FILE`인데 파일 없음 → 거부.
- `senderType` 값 `AI`는 내부적으로 `ASSISTANT`로 정규화된다.

## 6. `media` 도메인 — 업로드 & 생성 결과물 저장

### 6.1 구현되어 있는 부분

- `MediaStoragePort` (포트) / `S3MediaStorageAdapter` (유일한 구현체, AWS S3 SDK). 업로드 경로는 `users/{userId}/chats/{chatId}/input/{images|videos|files}/...` (사용자가 올린 원본) 또는 `.../generated/{images|videos}/...` (AI 생성 결과물)로 구분되며, 파일명은 `UUID.+ 확장자`, 상위 경로에 업로드 날짜(`LocalDate.now()`)가 들어간다.
- `FileUploadService.uploadChatFile()` — 채팅 메시지 첨부용 업로드 (`ChatMessageController`, `FileUploadController`에서 사용).
- `FileUploadService.uploadGeneratedChatFile()` — AI가 생성한 결과물(byte[])을 S3에 올리는 용도. **`GenerationResultService`에서만 호출**된다.
- `GeneratedAsset` / `AssetFile` 엔티티 + 리포지토리(Port/Adapter 패턴): AI 생성 결과(이미지/영상)와 그 결과에 딸린 실제 파일들을 저장. `GenerationResultService.saveAssistantAssetResult()`가 이 둘을 함께 저장하며, 채팅 메시지(`ChatMessage`, messageType=`ASSET`)와 1:1로 연결된다.

### 6.2 미구현 부분 — DTO만 존재

`MediaDtos.java`에 `CreateFolderRequest`, `StorageFolderResponse`, `SaveAssetRequest`, `SavedAssetResponse`, `DownloadResponse` 레코드가 정의되어 있고, DB 마이그레이션에도 `saved_folders`, `saved_media`, `downloads` 테이블이 존재한다. 그러나 이에 대응하는 `StorageFolder`/`SavedAsset`/`Download` **엔티티는 도메인 모델만 있고(`media/domain/model/`), 리포지토리·서비스·컨트롤러가 전혀 없다.** "생성 결과물을 폴더에 저장/즐겨찾기/다운로드 기록"이라는 라이브러리 기능은 스키마와 DTO 스켈레톤만 준비된 상태이고, 실제 동작하는 코드는 없다.

### 6.3 `GenerationResultService` — 호출하는 컨트롤러가 없음

가장 최근 머지된 기능 브랜치인 `feature/generated-result-storage`(커밋 `c2a65fb`)에서 추가된 `GenerationResultService`는 다음 기능을 완결된 형태로 제공한다.

- `saveAssistantTextResult(...)` — AI 텍스트 응답을 채팅 메시지로 저장하고 채팅방의 `isGenerating`을 끈다.
- `saveAssistantAssetResult(...)` — AI가 생성한 이미지/영상 결과 파일들을 S3 업로드 + `GeneratedAsset`/`AssetFile` 저장 + 채팅 메시지 생성까지 한 번에 처리.
- `startGenerating` / `finishGenerating` / `uploadGeneratedFile` 등 보조 메서드.

하지만 `generation/presentation/controller/` 패키지에는 `package-info.java`밖에 없고, 이 서비스를 호출하는 컨트롤러·스케줄러·다른 서비스가 **소스 전체에서 단 한 곳도 없다** (`grep` 결과 0건). 즉 이 기능은 백엔드 내부적으로 완성되어 있으나 **HTTP로 도달할 방법이 없는 죽은 코드**다. 아마도 AI 제공자(Claude/GPT/fal.ai) 연동 계층(7장)이 원래 이 서비스를 호출하도록 설계되었을 것으로 추정되나, 그 연동 계층 자체가 `dev`에는 없다.

> **2026-07-19 갱신**: 현재 작업 트리에는 7장에서 설명하는 AI 연동 계층(`GenerationAiService`, `GenerationController` 등)이 실제로 추가되었다. 그런데 `grep -r "GenerationResultService" src/main/java`로 다시 확인해도 여전히 참조하는 곳은 `GenerationResultService.java` 자기 자신뿐이다 — **새로 추가된 `GenerationController`도 `GenerationResultService`를 호출하지 않는다.** 새 컨트롤러는 `GenerationJob`을 저장/조회하는 별도의 흐름(Postman 테스트용, 7장 참고)이고, 그 결과를 채팅 메시지/`GeneratedAsset`으로 넘기는 다리는 아직 없다. 즉 이 절의 결론("호출하는 곳이 없는 죽은 코드")은 AI 연동 계층이 추가된 지금도 그대로 유효하다.

## 7. `generation` 도메인 — 스키마/엔티티만 존재, 사실상 미사용

`generation` 패키지에는 다음이 존재한다.

- `domain/model/{GenerationJob, GenerationQueue, GenerationStatus, GenerationType, AiModel}.java`
- `domain/repository/GenerationJobRepository.java` (Port 인터페이스)
- `infrastructure/persistence/{GenerationJobRepository(직접 JpaRepository 상속), GenerationJobRepositoryAdapter, SpringDataGenerationJobRepository}.java`
- `presentation/dto/GenerationJobDtos.java`
- `presentation/controller/package-info.java` (컨트롤러 없음)

`grep`으로 전체 소스를 확인한 결과, 이 중 **`GenerationJobDtos`, `GenerationQueue`, `AiModel`, `GenerationJob` 엔티티와 그 리포지토리/어댑터 일체를 실제로 사용(주입)하는 서비스나 컨트롤러가 전혀 없다.** DB 마이그레이션(V1/V2)에는 `generation_jobs`, `generation_queue`, `ai_models` 테이블이 정의되어 있어 스키마 레벨에서는 존재하지만, 애플리케이션 코드에서 이 테이블들에 실제로 읽기/쓰기를 수행하는 경로가 없다.

또한 `application.yml`에는 여전히 `app.ai.openai.api-key`, `app.ai.fal.api-key` 설정이 남아 있지만 이를 읽는 `@Value`/`@ConfigurationProperties` 코드가 없다 — 이 값들 역시 현재는 아무 의미가 없다.

**결론**: Claude 프롬프트 보정, GPT Vision 역프롬프트, fal.ai 이미지/영상 생성 제출·폴링 — 즉 이 프로젝트의 핵심 가치인 "AI 생성" 기능 자체를 수행하는 컨트롤러/서비스/외부 API 클라이언트가 `dev` 브랜치에는 **하나도 없다.** 있는 것은 그 결과를 "저장"하는 다운스트림 부분(`GenerationResultService`, 6.3절)뿐이며, 그마저도 호출할 곳이 없다.

이 기능은 `feature/api` 브랜치(커밋 `26dbd13 feat: initialize AI media studio backend`)에서 한 번 구현되었으나, **`dev`나 `main`으로 머지된 적이 없다** (`git log --all -- '*GenerationController*'` 결과 `feature/api` 계열 커밋에만 존재).

### 7.1 (2026-07-19 갱신) 현재 작업 트리에는 AI 연동 계층이 실제로 존재한다

위 7장 본문은 `dev`(`804ba1a`) 기준 서술이며 지금도 `dev` 자체에 대해서는 유효하다. 그러나 현재 작업 브랜치(`feature/api`, `dev` 병합 후 그 위에 스테이징된 변경 포함)에는 다음이 새로 추가되어 있다.

- **포트**: `PromptGenerationPort`(Claude), `ImageAnalysisPort`(GPT Vision), `FalGenerationPort`(fal.ai 큐 제출/폴링) — `application/port/out`.
- **실제 어댑터** (`@Profile("!local")`): `ClaudePromptGenerationAdapter`(`POST /v1/messages`, `anthropic-version` 헤더), `GptVisionImageAnalysisAdapter`(`POST /v1/chat/completions`, `image_url` data URL 방식), `FalGenerationAdapter`(`POST /{modelCode}` 제출, `GET /{modelCode}/requests/{requestId}/status` 폴링 — 슬래시가 포함된 `modelCode`를 URI 템플릿 변수가 아니라 문자열 결합으로 넘겨 `%2F` 인코딩 문제를 회피).
- **로컬용 Mock 어댑터** (`@Profile("local")`): `MockPromptGenerationAdapter`, `MockImageAnalysisAdapter`, `MockFalGenerationAdapter` — 실제 API를 호출하지 않고 즉시 가짜 결과를 반환해 비용 없이 로컬 개발이 가능하다. 9.5절에서 지적한 "로컬에서도 실제 자격증명이 필요하다"는 문제가 AI 쪽은 해소되었다(스토리지 쪽은 `MockMediaStorageAdapter` 추가로 마찬가지, 9.5절 갱신 노트 참고).
- **`GenerationAiService`**: "저장(PENDING/PROCESSING) → 트랜잭션 밖에서 외부 API 호출 → 결과 저장"을 각 메서드에서 개별 `save()`로 순차 처리한다(메서드 전체를 `@Transactional`로 묶지 않음 — 외부 호출 중 DB 커넥션을 잡고 있지 않기 위함). `generatePrompt`/`analyzeImage`/`submitFalJob`/`pollFalJob` 4개 유스케이스를 제공한다.
- **`GenerationController`** (`/api/v1/generation`, 인증 미적용): `POST /prompts`, `POST /reverse-prompts`, `POST /fal-jobs`, `GET /fal-jobs/{jobId}/status`. 컨트롤러 주석에 "Postman 통신 테스트용 엔드포인트"라고 명시되어 있으며, `userId`/`chatId`를 요청 바디로 직접 받는다(JWT `principal`을 쓰지 않음) — `트러블슈팅_가이드.md`가 과거에 지적했던 "임시 스펙"이 이 신규 엔드포인트에 그대로 재도입된 것이다.
- **`GenerationVideoPollingScheduler`**: 5초 간격(`@Scheduled(fixedDelay = 5000)`)으로 `QUEUED`/`PROCESSING` 상태인 이미지·영상 생성 작업을 폴링해, `COMPLETED`로 확인되면 fal.ai가 반환한 임시 미디어 URL을 `MediaStoragePort.storeFromUrl(...)`로 S3(또는 로컬 Mock)에 옮겨 담고 작업을 완료 처리한다.
- **`AiClientConfig`**: fal/Claude/OpenAI 각각의 `RestClient` 빈을 인증 헤더(`Key`, `x-api-key`, `Bearer`)와 함께 구성한다.

이로써 6.3절/7장 결론이 함의했던 "실행되면 `GenerationResultService`를 트리거할 AI 연동 계층"이 실제로 생겼다. 다만 **아직 그 다리는 없다** — `GenerationController`는 `GenerationJob`을 저장·조회할 뿐 `GenerationResultService.saveAssistantAssetResult(...)` 등을 호출해 채팅 메시지/`GeneratedAsset`으로 결과를 넘기지 않는다(6.3절 갱신 노트 참고). 이 계층이 추가된 커밋(병합 `ed77464`)이 한때 다른 파일들을 깨뜨려 브랜치가 컴파일조차 되지 않았으나, 2026-07-19 같은 날 모두 수정되어 지금은 컴파일이 통과한다 — 9.8절과 `트러블슈팅_가이드.md` 7~9장 참고.

## 9. 공통 인프라 & 발견된 문제점

이 장은 사용자 요청에 따라, 다른 브랜치와 비교할 때 반드시 참고해야 할 **`dev` 브랜치 자체의 문제점**을 모은 것이다.

### 9.1 (중요) 기존 `docs_h` 문서가 `dev`가 아닌 다른 브랜치를 기술하고 있음

`아키텍처_가이드.md`, `API_명세서.md`, `테스트_및_시뮬레이션_가이드.md`는 `GenerationController`, `GenerationAiService`, `AiClientConfig`, `ClaudePromptGenerationAdapter` 등 Claude/GPT Vision/fal.ai 연동 계층을 상세히 설명하지만, 이 클래스들은 `dev` HEAD 어디에도 존재하지 않는다. `git log --all`로 확인한 결과 이 파일들은 `feature/api` 브랜치(커밋 `26dbd13`, `2d415d0`)에만 존재했고 `dev`/`main`에 머지된 적이 없다. 즉 세 문서는 **`feature/api` 브랜치 기준으로 작성된 문서**이며, `dev`와 비교 자료로 쓰면 안 된다. (참고로 `docs_h/`, `docs_p/`는 git에 커밋되지 않은 로컬 파일이다 — `git status`에서 `??`로 표시됨.)

> **2026-07-19 갱신**: 현재 작업 브랜치가 `dev`를 병합한 `feature/api`로 바뀌면서, 위에서 "존재하지 않는다"고 한 `GenerationController`/`GenerationAiService`/`AiClientConfig`/`ClaudePromptGenerationAdapter`가 실제로 (스테이징된 상태로) 작업 트리에 생겼다. 즉 세 문서(`아키텍처_가이드.md` 등)의 서술 대상과 현재 코드가 다시 가까워졌다는 뜻이다. 다만 세부 스펙까지 일치하는지는 별도 확인이 필요하다(예: 세 문서가 인증된 JWT 기반 흐름을 전제로 하는지, 아니면 `GenerationController`처럼 `userId`를 요청 바디로 직접 받는 임시 스펙을 전제로 하는지). 병합 충돌로 한때 컴파일이 안 되던 문제(9.8절)는 2026-07-19에 수정되어, 이제는 세 문서에 나온 API를 실제로 호출해 검증할 수 있다.

### 9.2 `user` 도메인 내 리포지토리 접근 경로가 이중화되어 있고, 그중 하나는 죽은 코드

같은 `users` 테이블에 대해 두 개의 독립된 접근 경로가 공존한다.

1. `user.domain.repository.UserRepository`(Port) → `UserRepositoryAdapter` → `SpringDataUserRepository`(package-private). **`AuthService`, `AdminUserService`, `PasswordResetService`가 이 경로를 사용** — 실제 서비스 로직 전부.
2. `user.infrastructure.persistence.UserRepository`(Spring Data JPA 직접 상속). **`UserCommandService`만 이 경로를 사용**하며, `UserCommandService`는 `UserController`(`/api/v1/users`, `POST`로 `User.local()` 생성)에서만 호출된다.

`SecurityConfig`의 `permitAll` 목록에 `/api/v1/users`가 없으므로 이 엔드포인트는 `anyRequest().authenticated()`에 걸려 **인증 토큰 없이는 호출할 수 없다** — 그런데 회원가입 API가 로그인을 전제로 한다는 것 자체가 모순이다. 또한 `User.local()`로 만들어진 사용자는 `approvalStatus`/`accountStatus` 기본값이 각각 `PENDING`/`INACTIVE`로 세팅될 뿐 관리자 승인 플로우와 연결되지 않는다. 이 컨트롤러/서비스/DTO(`RegisterUserRequest`, `UserResponse`) 일체는 `AuthController`/`AuthService` 기반 회원가입(3.1절)이 자리잡기 전의 **레거시 코드로 보이며, 사실상 도달 불가능한 죽은 코드**다.

`generation` 도메인에서도 동일한 패턴(Port/Adapter 버전 + 직접 JpaRepository 버전이 같은 엔티티에 대해 공존)이 나타나지만, 7장에서 설명했듯 이쪽은 양쪽 다 사용처가 없다.

### 9.3 가장 최근 병합된 핵심 기능(`GenerationResultService`)이 호출 불가 상태

6.3절 참고. PR #29(`feature/generated-result-storage`)로 병합된 "AI 생성 결과 저장" 기능이, 이를 트리거해야 할 AI 연동 계층(7장) 없이 병합되어 현재 이 서비스에 도달하는 HTTP 경로가 전혀 없다. 다른 브랜치를 비교할 때 "이 기능이 실제로 동작하는지" 판단 기준으로 삼아야 한다.

### 9.4 에러 코드/예외 처리 설계가 도메인마다 다르고 일부는 미흡함

- `ErrorCode`는 코드 문자열 표기가 두 가지 스타일로 혼재한다: 초기 스타일(`COMMON-001`, `USER-001`, `GENERATION-001`, `PROVIDER-001` — 대시+번호)과 이후 추가된 스타일(`USER_LOGIN_ID_DUPLICATE`, `LOGIN_ID_NOT_FOUND`, `SIGNUP_PENDING` 등 — enum 이름 그대로). 게다가 `GENERATION_NOT_FOUND`, `INVALID_GENERATION_STATE`, `AI_PROVIDER_UNAVAILABLE`은 7장에서 설명한 대로 이제 아무도 던지지 않는 죽은 코드다.
- `ChatService.findOwnedChat()`, `ProjectService.findOwnedProject()`는 "대상 없음"과 "타인 소유/삭제됨"을 구분하지 않고 전부 `ErrorCode.INVALID_INPUT`(400)으로 처리한다. `user` 도메인은 `USER_NOT_FOUND`(404)처럼 상황별 코드를 세밀하게 나누는 반면, `chat`/`project`는 그렇지 않아 **같은 코드베이스 안에서 에러 응답 설계 원칙이 다르다.**
- `GlobalExceptionHandler`는 `BusinessException`과 `MethodArgumentNotValidException` 두 가지만 처리한다. FK 제약 위반(`DataIntegrityViolationException`), 잘못된 JSON 본문, 지원하지 않는 HTTP 메서드, 매핑되지 않는 경로(404) 등은 전부 Spring Boot 기본 에러 응답(`ApiResponse`/`ErrorResponse` 포맷이 아닌 형태)으로 새어나간다 — 프론트가 모든 에러를 동일한 포맷으로 파싱한다면 이 경로에서 깨질 수 있다.
  > **2026-07-19 갱신**: 현재 작업 트리의 `GlobalExceptionHandler`에는 `DataIntegrityViolationException`(제약 이름으로 `USER_NOT_FOUND`/`CHAT_NOT_FOUND` 등을 매핑) 핸들러와 `RestClientResponseException`(AI 공급자 비-2xx 응답 → `AI_PROVIDER_REQUEST_FAILED`) 핸들러가 추가되어 있다. 즉 이 항목에서 지적한 "FK 위반이 새어나간다"는 문제는 **`dev` 기준으로는 맞지만 현재 작업 트리에서는 이미 해결되어 있다.** 다만 잘못된 JSON 본문/지원하지 않는 HTTP 메서드/404 미매핑 경로는 여전히 처리되지 않는다.

### 9.5 로컬 개발 환경에서도 실제 AWS 자격증명이 필요함

`media` 도메인에는 `S3MediaStorageAdapter` 하나만 존재하고 `@Profile`로 분기되는 로컬 Mock 스토리지 어댑터가 없다(9.1절에서 언급한 `feature/api` 브랜치 기준 문서에는 "로컬 프로필용 더미 스토리지 어댑터"가 있다고 되어 있으나 `dev`에는 없음). 따라서 채팅 파일 업로드/생성 결과물 저장을 로컬에서 테스트하려면 `AwsConfig`가 `DefaultCredentialsProvider`로 실제 AWS 자격증명을 찾을 수 있어야 하고, `app.storage.bucket`도 실제 접근 가능한 버킷이어야 한다(로컬스택 등을 쓰려면 `STORAGE_ENDPOINT`/`STORAGE_PATH_STYLE_ACCESS`를 직접 맞춰야 함).

> **2026-07-19 갱신**: 현재 작업 트리에는 `media.infrastructure.storage.local.MockMediaStorageAdapter`(`@Profile("local")`)가 추가되어 있어 이 절의 문제가 구조적으로 해결되어 있다. `S3MediaStorageAdapter`(`!local` 프로필)와 `MediaStoragePort` 인터페이스가 병합 과정에서 한때 깨져 있었으나(9.8절), 2026-07-19에 `upload`/`storeFromUrl` 두 메서드로 정리하고 `MockMediaStorageAdapter`의 누락된 `upload` 구현도 추가해 `local`/`!local` 양쪽 프로필 모두 컴파일이 통과한다.

### 9.6 자동화 테스트가 사실상 없음

`src/test/java`에는 Spring 컨텍스트 로딩만 확인하는 `A1BackApplicationTests` 한 개뿐이다. 회원가입/로그인/채팅/파일업로드 등 핵심 비즈니스 로직에 대한 단위/통합 테스트가 없어, 다른 브랜치와 동작을 비교할 때 회귀 여부를 테스트 스위트로 확인할 수 없고 수동 검증에 의존해야 한다. (`테스트_및_시뮬레이션_가이드.md`가 존재하지만, 9.1절에서 설명했듯 이 문서 자체가 `dev`에는 없는 기능을 검증한다고 서술하고 있어 그대로 신뢰할 수 없다.)

### 9.7 사소한 설계 중복

`ChatController`에 `GET /api/chats?projectId={id}`와 `GET /api/projects/{projectId}/chats`가 동일한 서비스 메서드(`chatService.getChats`)를 서로 다른 경로로 중복 노출하고 있다. 기능상 문제는 없으나 API 표면이 불필요하게 넓다.

### 9.8 (2026-07-19 발견, 같은 날 해결) 병합 커밋(`ed77464`)이 최소 2개 파일을 컴파일 불가 상태로 깨뜨렸던 문제

`dev`를 `feature/api`에 병합한 커밋에서, 두 브랜치가 각각 수정한 파일이 충돌 마커 없이 뒤섞인 채로 커밋되어 `./gradlew compileJava`가 실패했었다. 상세 증상·원인·수정 내역은 `트러블슈팅_가이드.md` 7~9장에 정리했다. 요약:

- `global/exception/ErrorCode.java`: `AI_PROVIDER_UNAVAILABLE`/`USER_NOT_FOUND` enum 상수가 중복 선언되어 있던 것을 제거하고 하나의 상수 목록으로 합쳤다.
- `media/infrastructure/storage/s3/S3MediaStorageAdapter.java` / `MediaStoragePort` / `MockMediaStorageAdapter`: `feature/api` 버전(`upload(StorageUploadCommand)`)과 `dev` 버전(`store`/`storeFromUrl`)의 메서드 몸통이 서로 뒤섞여 있던 것을 확인 후, 실제로 아무도 호출하지 않던 `store(...)`를 제거하고 `upload`/`storeFromUrl` 두 메서드로 정리했다. `MockMediaStorageAdapter`에 빠져 있던 `upload` 구현도 추가했다.
- `GenerationJobDtos.Response.from(GenerationJob job)`: `Response` 레코드의 14개 필드(특히 `imageCategory`) 대비 인자를 13개만 넘기던 것을 고쳤다(`트러블슈팅_가이드.md` 8장).
- `GenerationJob` 엔티티: DB 마이그레이션(`V4`)이 `job_type` 컬럼을 `generation_type`으로 리네임했는데도 `create()` 팩토리 메서드와 일부 호출부가 옛 이름(`jobType`/`getJobType()`)을 참조하던 것을 `generationType`/`getGenerationType()`으로 통일했다(`트러블슈팅_가이드.md` 9장).

위 네 가지를 모두 고친 뒤 `./gradlew compileJava`와 `./gradlew compileTestJava`가 에러 없이 통과하는 것을 확인했다. **이 문서의 다른 장에서 "정적 분석만으로 확인했다"고 적은 부분들은 이제 최소한 컴파일은 되는 코드를 대상으로 한다** — 다만 실제 런타임 동작(DB/외부 API 연동)까지 검증한 것은 아니라는 점은 여전히 유의할 것.

## 10. 다른 브랜치와 비교할 때 확인할 체크리스트

1. **AI 생성 연동 계층 존재 여부** — Claude/GPT Vision/fal.ai 호출 코드, 그리고 이를 `GenerationResultService`(또는 대체 서비스)와 연결하는 컨트롤러가 있는가? (`dev`에는 없음, 7장)
2. **`user` 도메인 리포지토리 경로가 단일화되어 있는가**, 아니면 `dev`처럼 이중화된 죽은 코드가 남아있는가? (9.2절)
3. **`media` 도메인의 "저장/폴더/다운로드" 기능이 실제 구현되어 있는가**, 아니면 `dev`처럼 DTO/스키마만 있는가? (6.2절)
4. **에러 코드 네이밍/처리 범위가 일관되어 있는가** — 특히 FK 위반이나 예상치 못한 예외를 `ApiResponse` 포맷으로 감싸는 전역 핸들러가 추가되어 있는가? (9.4절)
5. **로컬 개발용 Mock 스토리지/메일 어댑터**가 `@Profile("local")` 등으로 분리되어 있는가? (9.5절)
6. **테스트 커버리지** — 비즈니스 로직에 대한 단위/통합 테스트가 추가되어 있는가? (9.6절)
7. **(2026-07-19 추가) 브랜치가 실제로 컴파일/실행되는가** — `./gradlew compileJava`를 먼저 돌려서 확인할 것. `feature/api`는 한때 병합 충돌 잔재로 컴파일 자체가 실패했으나(9.8절, `트러블슈팅_가이드.md` 7~9장) 2026-07-19에 수정되어 현재는 통과한다. 다른 브랜치를 비교할 때도 코드를 눈으로만 읽고 "구현되어 있다"고 판단하지 말고 빌드가 통과하는지 반드시 확인할 것 — 이번처럼 병합 직후에는 특히 깨져 있을 수 있다.
