# 로컬 개발 환경 설정 가이드 (온보딩)

새로 합류한 개발자가 `a1-back`을 로컬 PC에서 처음부터 기동하기 위한 단일 매뉴얼입니다.

---

## 1. 필요한 인프라

```bash
docker compose up -d   # PostgreSQL(5433), Redis(6379), MinIO(9000/9001)
```

전부 `healthy` 상태인지 확인 후 다음 단계로 진행합니다.

### ⚠️ Windows에서 흔한 문제 — 로컬 Postgres와 포트 충돌

이 프로젝트 컨테이너는 Postgres를 **5433** 포트로 노출하지만, PC에 네이티브(비-Docker) PostgreSQL이 이미 **5432** 포트에서 떠 있는 경우가 있습니다(Windows 설치형 `postgres.exe` 등). 이 경우 두 프로세스가 뒤섞여 "password 인증 실패"처럼 보이는 에러가 날 수 있습니다 — **코드 문제가 아니라 PC 환경 문제**입니다.

`netstat`으로 5432에 여러 PID가 붙어 있는지 먼저 확인하고, 다음 중 하나로 해결합니다:
1. Windows 서비스 관리자(`services.msc`)에서 로컬 PostgreSQL 서비스를 이 프로젝트 작업 중엔 중지.
2. `compose.yml`은 건드리지 않고, 본인 PC의 `.env`에서 `DB_URL`만 다른 포트(`localhost:5433`)를 가리키도록 개인 설정.

---

## 2. 서버 기동

```bash
./gradlew bootRun   # 8080 포트, 프로필 미지정 시 기본값 local
curl http://localhost:8080/actuator/health
```

기동 로그에서 Flyway가 `V1__initial_schema.sql`, `V2__align_schema_with_latest_erd.sql`을 순서대로 적용하고 `Hibernate ddl-auto=validate`가 통과하는지 확인합니다. 정상 기동 시 `users`, `chats`, `ai_models`, `generation_jobs`(구 `generation_queue`), `saved_folders`, `saved_media`, `downloads`, `notifications`, `email_verifications`, `password_reset_tokens` 등 모든 테이블이 Entity와 1:1로 맞아떨어집니다.

기본 프로필은 `local`이며, 이 상태에서는 Claude/GPT Vision/fal.ai 3개 AI 공급자가 전부 **Mock 어댑터**로 동작해 외부 API 비용이 전혀 발생하지 않습니다. Mock 동작 방식과 실제 API 전환 방법은 `테스트_및_시뮬레이션_가이드.md`를 참고하세요.

---

## 3. 테스트용 유저/채팅방 시딩

`generation_jobs.user_id`, `generation_jobs.chat_id`는 `users`/`chats`에 대한 `NOT NULL` FK이기 때문에, 빈 DB 상태에서는 어떤 생성 API 요청도 무조건 실패합니다. **현재 회원가입 API(`POST /api/v1/users`)가 500 버그로 막혀 있어** (`트러블슈팅_가이드.md` 참고) 테스트 유저는 SQL로 직접 넣어야 합니다.

```sql
INSERT INTO users (login_id, email, password_hash, name, birth_date, phone_number, created_at, updated_at)
VALUES ('tester', 'tester@example.com', 'hash', 'Tester', '1990-01-01', '010-0000-0000', now(), now())
RETURNING id;

INSERT INTO chats (user_id, title, is_generating, created_at, updated_at)
VALUES (<위에서 나온 id>, 'Test chat', false, now(), now())
RETURNING id;
```

**주의**: 생성되는 `id`는 그날그날의 시퀀스 상태에 따라 달라집니다. `userId`/`chatId`를 하드코딩하지 말고, 매번 아래 쿼리로 실제 존재하는 값을 확인한 뒤 사용하세요.

```sql
select id from users order by id desc limit 1;
select id from chats where user_id = <위 id> order by id desc limit 1;
```

DB를 리셋(`docker compose down -v`)하거나 다른 팀원 PC에서 돌리면 id가 또 바뀝니다.

## 4. 환경 변수 (`.env`)

| 변수 | 용도 |
|---|---|
| `DB_URL`, `DB_USERNAME`, `DB_PASSWORD` | PostgreSQL 연결 |
| `REDIS_HOST`, `REDIS_PORT` | Redis 연결 |
| `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, `STORAGE_BUCKET`, `STORAGE_ENDPOINT`, `STORAGE_PUBLIC_BASE_URL` | 로컬은 MinIO, 운영은 실제 S3 |
| `ANTHROPIC_API_KEY`, `CLAUDE_MODEL`(기본 `claude-sonnet-5`), `ANTHROPIC_VERSION`(기본 `2023-06-01`) | Claude Messages API |
| `OPENAI_API_KEY`, `OPENAI_VISION_MODEL`(기본 `gpt-4.1`) | GPT Vision (Chat Completions) |
| `FAL_API_KEY` | fal.ai 큐 API |

`local` 프로필에서는 위 AI 관련 키가 없어도 Mock 어댑터가 동작하므로 개발을 시작하는 데 지장이 없습니다.

### 🚨 심각한 보안 이슈 — 실제 키가 코드/템플릿에 평문으로 커밋되어 있음 (미해결)

`.env.example`과 `src/main/resources/application.yml`의 `${ANTHROPIC_API_KEY:...}` / `${OPENAI_API_KEY:...}` / `${FAL_API_KEY:...}` **기본값 자리에 실제 Anthropic/OpenAI/fal.ai API 키 값이 그대로 박혀 있습니다.** `.env`를 만들지 않아도 이 기본값으로 실제 과금 호출이 나갈 수 있고, `.env.example`은 템플릿이라 커밋되는 파일이라 저장소 히스토리에 키가 영구히 남습니다.

- 이 문제는 최소 2026-07-13(day5)부터 인지되었으나 2026-07-15 기준 아직 미해결 상태입니다.
- 조치 필요 사항(코드 소유자/팀 판단 필요, 자세한 절차는 `트러블슈팅_가이드.md` 참고):
  1. 세 키 전부 폐기(revoke) 후 재발급.
  2. `.env.example`/`application.yml`의 기본값을 빈 문자열/플레이스홀더로 되돌리기.
  3. 실제 키는 `.env`(gitignore 대상)에만 보관.

## 5. Mock ↔ 실제 API 전환

```bash
./gradlew bootRun                          # Mock (기본값, 아무 설정 없이 실행)
SPRING_PROFILES_ACTIVE=prod ./gradlew bootRun  # 실제 Claude/GPT Vision/fal.ai 호출
```

`prod` 프로필이어도 `DB_URL`/`REDIS_HOST` 등은 `.env` 값을 그대로 쓰므로, DB는 로컬 Postgres를 계속 보게 하면서 AI 어댑터만 실제로 전환해 검증할 수 있습니다. 상세 내용과 비용 통제 전략은 `테스트_및_시뮬레이션_가이드.md`를 참고하세요.
