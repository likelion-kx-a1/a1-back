## 1. `generation_jobs_user_id_fkey` / `chat_id_fkey` 위반 (FK 오류)

### 증상

```
ERROR: insert or update on table "generation_jobs" violates foreign key constraint "generation_jobs_user_id_fkey"
Detail: Key (user_id)=(1) is not present in table "users".
```

### 원인

`GenerationAiService`는 "저장(PROCESSING) → 외부 API 호출 → 결과 저장" 순서로 동작합니다(`아키텍처_가이드.md` 5장 참고). 이 에러는 **1단계(최초 INSERT)에서 막힌 것**이라, Claude/GPT/fal.ai API 키가 맞는지는 아직 검증되지도 않은 단계입니다. **AI 연동 문제가 아닙니다.**

`userId`/`chatId`는 JWT 인증 및 chat 생성 API가 없어 요청 바디로 직접 받는 임시 스펙이라, 존재하지 않는 값(특히 예시 문서에 박혀 있는 `userId: 1`, `chatId: 1` 같은 고정값)을 그대로 복붙하면 이 에러가 나기 쉽습니다.

### 해결

DB에 실제로 존재하는 조합을 매번 조회해서 사용합니다.

```sql
select id from users order by id desc limit 1;
select id from chats where user_id = <위 id> order by id desc limit 1;
```

값을 하드코딩하지 마세요 — DB를 리셋(`docker compose down -v`)하거나 다른 팀원 PC에서 실행하면 id가 달라집니다. 테스트 데이터 시딩 방법은 `로컬_개발환경_설정_가이드.md` 3장을 참고하세요.

### 응답 포맷 이력 (수정 완료)

- **2026-07-12 시점**: 존재하지 않는 `userId`로 요청하면 500 Internal Server Error + Spring 기본 whitelabel 포맷(`{timestamp,status,error,path}`)이 그대로 노출되는 문제가 있었습니다. `GlobalExceptionHandler`가 `BusinessException`/`MethodArgumentNotValidException`/`RestClientResponseException`만 처리하고 `DataIntegrityViolationException`은 잡지 못했기 때문입니다.
- **2026-07-14 회귀 테스트 시점**: 동일 케이스가 **404 `USER-002`(존재하지 않는 사용자입니다)**, 프로젝트 표준 `ErrorResponse` 포맷으로 정상 응답하는 것을 확인했습니다. `GlobalExceptionHandler`에 FK 위반 → `USER-002`/`CHAT-001` 매핑이 추가된 것으로 보입니다. 현재 스펙은 `API_명세서.md` 부록 A를 참고하세요.

## 2. 회원가입 500 버그 (`POST /api/v1/users`) — 미해결

### 증상

`POST /api/v1/users`로 정상 회원가입 시도 시 500 에러.

### 원인

`V2__align_schema_with_latest_erd.sql`이 `users.birth_date`, `users.phone_number`를 `NOT NULL`로 만들었는데, `User.local(email, passwordHash, name)` 팩토리 메서드와 `RegisterUserRequest`가 이 두 필드를 전혀 채우지 않습니다 → INSERT 시 NOT NULL 제약 위반.

### 현재 상태

JWT/인증을 담당하는 팀원의 작업 범위와 겹쳐 **의도적으로 코드를 고치지 않고 기록만 해둔 상태**입니다. 이 버그 때문에 현재 DB의 테스트 유저는 전부 API가 아니라 SQL로 직접 INSERT한 값입니다(`로컬_개발환경_설정_가이드.md` 3장 참고). 이 버그가 해결되기 전까지는 "userId를 요청 바디로 그대로 받아 신뢰 없이 저장"하는 임시 구조가 계속 남아있고, 그때마다 1장의 FK 불일치가 재발할 수 있습니다.

## 3. Windows 로컬 환경 — Postgres 포트 충돌로 인한 "password 인증 실패"

PC에 Docker와 별개로 네이티브 PostgreSQL이 5432 포트에서 이미 떠 있으면, `docker compose up -d`로 컨테이너를 띄워도 앱이 `localhost:5432`로 붙을 때 어느 쪽이 응답하는지 꼬여서 password 인증 실패가 납니다. **코드 문제가 아니라 PC 환경 문제**이며, 해결 방법은 `로컬_개발환경_설정_가이드.md` 1장을 참고하세요.

## 4. `.env` 파싱 실패 — `key cannot contain a space`

### 증상

`docker compose up -d` 실행 시 `.env` 파일 파싱 에러로 컨테이너 기동 자체가 실패.

### 원인

`.env`에 `claude api = sk-ant-...`처럼 **공백이 포함된 잘못된 키 이름**이 들어있었습니다. 올바른 변수명은 `ANTHROPIC_API_KEY=...`입니다. 이 오타 때문에 두 가지 문제가 동시에 발생합니다:
1. 공백 때문에 `.env` 파일 자체 파싱이 깨져 Docker Compose가 실패.
2. 설령 파싱이 통과해도 `ANTHROPIC_API_KEY`라는 이름으로 안 읽혀서, 실제로는 `application.yml`에 하드코딩된 (유출된) 기본 키로 계속 호출이 나가고 있었음 (아래 5장 참고).

### 해결

`.env`의 해당 줄을 `ANTHROPIC_API_KEY=sk-ant-...` 형식으로 수정합니다. **`.env.example`에도 동일한 오타가 남아있으니 팀 차원에서 같이 고쳐야 합니다.**

### 상태 (2026-07-19 확인 — 해결됨)

현재 `.env.example`에는 공백이 들어간 오타가 더 이상 없습니다. 다만 `.env.example`에 `OPENAI_API_KEY`, `FAL_API_KEY`는 있지만 **`ANTHROPIC_API_KEY` 항목 자체가 빠져 있습니다.** `application.yml`이 `${ANTHROPIC_API_KEY:}`로 기본값을 빈 문자열 처리하기 때문에 앱이 죽지는 않지만, Claude 어댑터(`ClaudePromptGenerationAdapter`, 7장 참고)가 빈 키로 호출해 401을 받게 되므로 `.env.example`에 항목을 추가해두는 게 좋습니다.

## 7. 🚨 `dev` 병합(`ed77464`) 이후 브랜치가 아예 컴파일되지 않음 — 해결됨 (2026-07-19 발견, 같은 날 수정)

### 증상

```
> Task :compileJava FAILED
ErrorCode.java:16: error: enum constant not expected here
S3MediaStorageAdapter.java:55: error: illegal start of expression
S3MediaStorageAdapter.java:79: error: ';' expected
S3MediaStorageAdapter.java:84: error: ';' expected / not a statement
S3MediaStorageAdapter.java:131: error: illegal start of expression
9 errors
BUILD FAILED
```

`./gradlew compileJava`가 즉시 실패합니다. AI 연동이나 DB 문제가 아니라 **자바 문법 자체가 깨진 상태**라 어떤 기능도 테스트할 수 없습니다.

### 원인

`dev`를 `feature/api`에 병합한 커밋(`ed77464 Merge branch 'dev' into feature/api`)에서, 두 브랜치가 각각 독립적으로 수정한 파일이 충돌 마커 없이 서로 다른 버전 내용이 섞인 채로 커밋되었습니다(수동 충돌 해결 과정에서 정리가 덜 된 것으로 추정).

1. **`ErrorCode.java`** — `AI_PROVIDER_UNAVAILABLE` enum 상수가 7번째 줄(13행)과 그 직후(16행)에 **두 번 선언**되어 있고, 15행에서 enum 상수 목록이 세미콜론(`;`)으로 이미 닫힌 뒤 16행에 또 다른 enum 상수 목록이 이어져 있습니다. `feature/api`가 쓰던 버전(`GENERATION-*`/`PROVIDER-*` 코드)과 `dev`가 쓰던 버전(`USER_LOGIN_ID_DUPLICATE` 등 이름 그대로 쓰는 코드)이 둘 다 살아남은 채 이어붙여진 형태입니다.
2. **`S3MediaStorageAdapter.java`** — `feature/api` 버전(`upload(StorageUploadCommand command)` 메서드, `StorageUploadResult` 반환)과 `dev` 버전(`store(byte[], ...)` / `storeFromUrl(...)` 메서드)이 같은 메서드 바디 안에서 뒤섞여 있습니다. 예를 들어 55행에서 `upload()` 메서드 바디 중간에 `public String store(...)` 선언이 그대로 끼어들고, 79행 근처에는 `resolvePublicUrl()`의 `return`문 중간에 `RequestBody.fromBytes(content));`가 끼어 있는 등 두 메서드가 서로의 몸통 안으로 잘려 들어간 상태입니다.
3. 그 여파로 **`MediaStoragePort` 인터페이스도 분기 이전 두 버전의 메서드(`upload`, `store`, `storeFromUrl`)를 전부 합친 3개짜리 인터페이스**가 되어 있는데, `FileUploadService`는 `upload(...)`만 호출하고 `GenerationVideoPollingScheduler`(신규 스테이징 파일)는 `storeFromUrl(...)`만 호출합니다. `MockMediaStorageAdapter`(local 프로필용)는 `store`/`storeFromUrl`만 구현하고 `upload`는 구현하지 않아, **컴파일 에러가 해결되더라도 곧바로 "MockMediaStorageAdapter가 추상 메서드 upload를 구현하지 않았다"는 다음 에러가 날 것**입니다.

### 해결 (완료 — 2026-07-19)

자동 병합으로 복구할 수 없어 각 파일에서 어느 브랜치의 의도를 살릴지 직접 확인하고 고쳤습니다.

- **`ErrorCode.java`**: 중복된 `AI_PROVIDER_UNAVAILABLE`, `USER_NOT_FOUND` 선언을 제거하고 enum 상수 목록을 세미콜론 하나로 다시 합쳤습니다. `USER_NOT_FOUND`는 기존 `USER-002`(대시+번호 스타일, `GlobalExceptionHandler`의 FK 매핑과 1장에서 참조하던 값)를 그대로 유지했습니다.
- **`S3MediaStorageAdapter.java` / `MediaStoragePort`**: 호출부를 먼저 확인해보니 `FileUploadService`는 `upload(StorageUploadCommand)`만, `GenerationVideoPollingScheduler`는 `storeFromUrl(...)`만 쓰고 있었고 `store(byte[], ...)`는 실제로 아무 곳에서도 호출되지 않는 죽은 메서드였습니다. 그래서 `MediaStoragePort`에서 `store(...)`를 제거하고 `upload`/`storeFromUrl` 두 개로 정리한 뒤, `S3MediaStorageAdapter`를 두 메서드가 섞이지 않도록 다시 작성했습니다.
- **`MockMediaStorageAdapter`**: `upload(StorageUploadCommand)` 구현이 빠져 있던 것을 추가해 `MediaStoragePort`를 완전히 구현하도록 했습니다.
- 위 세 파일을 고친 뒤 `./gradlew compileJava`를 다시 돌려 8장·9장에서 설명하는 후속 에러들을 추가로 확인·수정했습니다.

## 9. `GenerationJob` 엔티티 필드명 불일치(`jobType` vs `generationType`) — 해결됨 (2026-07-19 발견, 같은 날 수정)

### 증상

7·8장을 고친 뒤 다시 빌드하자 이번엔 아래 에러가 났습니다.

```
GenerationJob.java:73: error: cannot find symbol
    job.jobType = jobType;
       ^
  symbol:   variable jobType
GenerationVideoPollingScheduler.java:51: error: cannot find symbol
    GenerationType type = parseType(job.getJobType());
                                       ^
  symbol:   method getJobType()
GenerationJobDtos.java:42: error: cannot find symbol
          job.getJobType(),
             ^
  symbol:   method getJobType()
4 errors
```

이전 세 장의 파싱/인자 개수 에러 때문에 javac가 이 파일까지 도달하지 못해 가려져 있던, 별개의 병합 잔재 버그입니다.

### 원인

DB 마이그레이션 `V4__rename_media_to_asset_schema.sql`에서 `generation_jobs.job_type` 컬럼을 `generation_type`으로 리네임했습니다. `Chat`/`ChatMessage` 엔티티는 이미 `generationType` 필드 + `getGenerationType()`으로 이 리네임을 반영하고 있었지만, `GenerationJob` 엔티티만 컬럼명은 `generationType`(29행 필드 선언)으로 바꿔놓고 **`create()` 팩토리 메서드 안에서는 여전히 존재하지 않는 `jobType` 필드에 대입하고 있었습니다** (`job.jobType = jobType;`). 이 때문에 Lombok이 생성하는 게터도 `getGenerationType()`이지 `getJobType()`이 아닌데, `GenerationVideoPollingScheduler`와 `GenerationJobDtos.Response.from()`은 옛 이름(`getJobType()`)을 그대로 참조하고 있었습니다.

### 해결

`GenerationJob.create()`의 `job.jobType = jobType;`를 `job.generationType = jobType;`로 고치고, `GenerationVideoPollingScheduler`(2곳)와 `GenerationJobDtos.Response.from()`의 `job.getJobType()` 호출을 전부 `job.getGenerationType()`으로 바꿨습니다. 다른 도메인(`Chat`/`ChatMessage`)과 이름 규칙이 통일되었습니다.
